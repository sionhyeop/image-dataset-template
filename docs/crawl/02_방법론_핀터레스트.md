# Pinterest 기반 대량 이미지 레퍼런스 수집 파이프라인 요구사항 정리 (v2)

v1 대비 변경 요약은 3장에 있다. 카테고리 체계, 검색어 목록, 폴더 구조, 검수 흐름은 v1을 유지하되, 수집 방식과 점수화 설계를 실측 가능한 형태로 교체했다.

---

## 1. 배경

현재 프로젝트에서는 팀원 간 이해를 돕기 위해, 인물 사진의 구도/포즈/장소/무드별 레퍼런스 이미지셋이 필요하다.

무료 이미지 스톡 사이트(Unsplash, Pexels 등)는 광고성 스톡 이미지 비중이 높아, Pinterest에서 보이는 감성 스냅(카페샷, 해변샷, 야경 보케, 로우앵글)과 분위기가 다르다. 필요한 것은 촬영 구도를 참고할 수 있는 고관련도 인물 구도 레퍼런스다.

이번 목적은 상용 학습용 권리 검증 데이터셋 구축이 아니라, **팀원 이해용 구도 레퍼런스 이미지셋을 빠르게 대량으로 구성하는 것**이다. 수집물의 용도는 내부 열람으로 한정한다. AI 학습 투입 여부는 별개 사안이며, 필요해지는 시점에 17장 기준으로 별도 검토한다.

---

## 2. 사용자 요구사항 요약

1. 카테고리별로 Pinterest에서 관련도 높은 이미지를 자동으로 많이 모은다.
2. 사람이 보드에 일일이 저장하는 방식은 너무 느리므로, 선다운로드 후검수로 진행한다.
3. 다운로드된 이미지는 카테고리별로 정리되어야 한다.
4. 중복 제거, 저해상도 제거, contact sheet 생성까지 자동화한다.
5. 최종 판단은 사람이 한다.

## 3. v1 검증 결과 및 설계 변경 사항

v1 계획을 그대로 구현할 경우 실제로 막히는 지점과, 문서 내부 결함을 다음과 같이 반영했다.

```txt
[막히는 지점]
1. 비로그인 상태에서는 검색 결과 초기 로드가 25~50장 수준이고,
   조금만 스크롤하면 닫을 수 없는 로그인 벽이 뜬다.
   → 로그인된 세션(쿠키 또는 persistent context 최초 1회 수동 로그인)이
     대량 수집의 전제 조건. 지시문에 명시.

2. DOM srcset의 실질 최대 해상도는 736x다.
   URL 크기 세그먼트를 /originals/로 치환하는 트릭은
   2025년 이후 403 반환이 늘어 신뢰할 수 없다.
   → 내부 API 기반 수집은 원본 해상도 URL을 JSON으로 직접 받으므로
     API 기반 도구를 1순위로 사용.

3. Playwright 수집기를 직접 구현할 필요가 없다.
   pinterest-dl(PyPI, Python 3.10+)이 검색 수집, 핀+연관핀 수집,
   브라우저 쿠키 로그인, JSON 메타데이터 출력을 이미 지원한다.
   → v1의 01/02 스크립트 역할 대부분을 대체. 직접 구현은 폴백으로 강등.

[문서 내부 결함 수정]
4. 글로벌 중복 제거가 4축 분류와 충돌한다.
   같은 사진이 WHERE/POSE/FRAME/STYLE 여러 축에 정당하게 속할 수 있다.
   → 중복 제거는 카테고리 내부만. 교차 축 중복은 삭제하지 않고
     multi_label로 기록.

5. duplicate_penalty가 final_score 계산식에 있었으나,
   중복은 03단계에서 이미 제거된 뒤다. → 점수식에서 제거.

6. alt 텍스트 기반 인물 판정은 alt가 비어 있거나 자동 생성인 경우가
   많아 취약하다. → 경량 person detector(YOLOv8n)로 교체,
   alt는 보조 신호로만 사용.

7. CLIP은 "가능하면"이 아니라 확정 채택. open_clip ViT-B/32 기준
   1.5만 장 규모는 Apple Silicon MPS에서 수십 분 수준.
   카테고리 프롬프트 전체와의 zero-shot 비교로 wrong_category
   자동 플래그까지 생성.

8. 4.4의 S06_natural_light_beach가 8.5 검색어 목록에서 누락 → 추가.

9. queries.csv의 target_count와 CLI --max-per-query가 이중 정의
   → CSV의 target_count를 기본값으로, CLI 플래그는 override로 정리.
```

## 5. 수집 방식 선택 (v2 핵심 변경)

세 가지 방식을 정의하고, A를 기본으로 B를 보강용으로 조합한다.

### 방식 A — pinterest-dl 검색 수집 (기본)

pinterest-dl의 API 클라이언트로 검색어별 수집을 수행한다. 브라우저 없이 동작하고, 원본 해상도 URL과 alt 텍스트가 포함된 메타데이터를 JSON으로 받을 수 있다.

```bash
pip install pinterest-dl        # Python 3.10+
pinterest-dl login --from-browser -o config/cookies.json
pinterest-dl search "cafe aesthetic portrait pose" -n 300 --json -c config/cookies.json
```

로그인 쿠키 없이도 일부 수집은 되지만, 검색어당 300장 이상을 안정적으로 확보하려면 쿠키를 전달한다.

### 방식 B — 시드 핀 기반 연관핀 수집 (정밀도 보강)

검색 수집은 뒤로 갈수록 관련도가 떨어진다. 이를 보완하기 위해, 카테고리마다 사람이 마음에 드는 대표 핀 10~20개를 골라 `config/seed_pins.csv`에 넣고, 각 시드 핀의 연관핀을 수집한다. Pinterest 추천엔진이 유사 구도/무드 이미지를 골라주므로 검색보다 카테고리 적합도가 높다.

```bash
pinterest-dl scrape "https://www.pinterest.com/pin/<id>/" -n 50 --json -c config/cookies.json
# 핀 1개 + 연관핀 49개
```

시드 선정에 드는 사람 시간은 카테고리당 5~10분 수준으로, v1이 배제했던 수동 큐레이션과 달리 부담이 작다. 관련도가 특히 중요한 카테고리(POSE, STYLE 축)에 우선 적용한다.

### 방식 C — Playwright DOM 수집 (폴백)

pinterest-dl은 비공식 도구라 Pinterest 내부 API 변경 시 일시적으로 깨질 수 있다. 그 경우를 대비해 v1의 Playwright 방식을 폴백으로 유지하되, 다음 두 가지를 전제 조건으로 명시한다.

```txt
1. persistent context 프로필에 최초 1회 수동 로그인 필수.
   비로그인 상태에서는 로그인 벽 때문에 검색어당 수십 장 이상 수집 불가.
2. 해상도 상한은 736x로 간주.
   srcset 최대값 선택 후, /originals/ 치환은 시도하되 403 시
   1200x → 736x 순으로 폴백.
```

pinterest-dl 자체도 `pip install pinterest-dl[browser]` 후 `--client chromium`으로 브라우저 백엔드를 제공하므로, 순수 자작 Playwright 구현은 최후 수단이다(부록 참고).

### 권장 조합

```txt
1차: 방식 A로 전 카테고리 넓게 수집 (검색어당 300~500)
2차: 결과가 빈약하거나 관련도가 낮은 카테고리에 방식 B 적용
     (카테고리당 시드 15개 × 연관 50 = 약 750 후보 추가)
```

## 9. 파이프라인 설계

```txt
queries.csv / seed_pins.csv 작성
→ 01_collect: pinterest-dl로 후보 메타데이터 수집 (search_results.csv)
→ 02_download: URL 정규화로 사전 중복 제거 후 다운로드 (downloads.csv)
→ 03_deduplicate: pHash 카테고리 내 중복 제거 + 교차 축 멀티라벨 기록
→ 04_rank: person detector + CLIP + 해상도/비율 점수 (review_queue.csv)
→ 05_contact_sheets: 카테고리별 contact sheet + HTML 검수 갤러리
→ 사람이 최종 검수
```

### 9.1 image_id와 URL 정규화 (결정적 식별자)

pinimg URL은 크기 프리픽스만 다르고 뒷 경로가 같으면 동일 이미지다.

```txt
https://i.pinimg.com/236x/ab/cd/ef/abcdef123.jpg
https://i.pinimg.com/736x/ab/cd/ef/abcdef123.jpg
→ 정규화 경로: ab/cd/ef/abcdef123.jpg
→ image_id = sha1(정규화 경로)[:12]
```

이 규칙으로 다운로드 전에 URL 수준 중복(리핀으로 인한 동일 이미지)을 카테고리 내에서 제거해 대역폭을 아끼고, resume과 재실행이 결정적으로 동작한다. 같은 이미지가 서로 다른 카테고리에 나타나는 경우는 각각 다운로드하고(디스크 비용 미미) 03단계에서 멀티라벨로 연결만 한다.

### 9.2 다운로드 규칙

```txt
- API가 준 원본 URL을 1순위로 시도
- 403/404 시 1200x → 736x 순 폴백
- User-Agent와 Referer(https://www.pinterest.com/) 헤더 설정
- 동시성 4, 요청 간 0.3~0.8초 랜덤 지터
- min_width/min_height 미달은 status=too_small
- 실패는 errors.csv에 사유와 함께 기록, --resume 지원
```

### 9.3 예의 있는 수집 (v1 제한 유지)

```txt
- 로그인 우회, 캡차 우회, 프록시 회전, 차단 회피 기능은 만들지 않는다.
- 쿼리 간 5~10초 랜덤 대기.
- 한 세션에 몰아 돌리지 말고, 실패율이 오르면 중단 후 나중에 재개.
- 계정 제한 가능성은 감수해야 하는 리스크임을 인지한다 (17장).
```

## 10. 자동 점수화 기준 (v2 개정)

중복은 03단계에서 이미 제거되었으므로 점수식에 포함하지 않는다. 비율은 width/height로 정의한다.

```txt
person_score (YOLOv8n person 클래스):
- 사람 1명 검출 +2 (1인 구도 레퍼런스 우선)
- 사람 2명 이상 +1
- 미검출 -3 (사실상 검수 후순위로 밀림)

clip_score (open_clip ViT-B/32):
- 이미지와 해당 query 텍스트의 cosine similarity
- 카테고리 내부에서 z-score 정규화 후 -2 ~ +2로 클리핑
- 동시에 전체 카테고리 프롬프트와 zero-shot 비교하여
  argmax가 배정 카테고리와 다르면 category_match=False 플래그

resolution_score:
- 긴 변 1200px 이상 +2
- 긴 변 800px 이상 +1
- 500px 미만 -2

aspect_score (w/h):
- 0.60~0.85 세로형 +2
- 0.85~1.10 +1
- 1.30 초과 가로형 -1

keyword_score (보조 신호):
- alt 또는 URL에 query 키워드 포함 +1

final_score =
  2.0 * clip_score + person_score + resolution_score
  + aspect_score + keyword_score
```

CLIP과 YOLO는 requirements에서 optional 그룹으로 분리하고, 미설치 환경에서는 해당 항을 0으로 두고 나머지 점수만으로 동작해야 한다.

## 11. Claude Code에 전달할 최종 구현 지시문

아래 내용을 Claude Code에게 그대로 전달한다.

```txt
Pinterest 검색 기반 대량 이미지 수집 파이프라인을 만들어줘.

목표:
팀원 이해용 구도 레퍼런스 이미지셋을 만들기 위해, 카테고리별 검색어와
시드 핀을 기준으로 Pinterest에서 이미지를 자동 수집/다운로드하고,
중복 제거와 점수화를 거쳐 사람이 검수할 수 있는 형태로 정리한다.

중요 제한:
- 로그인 우회, 캡차 우회, 프록시 회전, 차단 회피 기능은 만들지 않는다.
- 쿼리 간 5~10초 랜덤 대기, 다운로드 요청 간 0.3~0.8초 지터를 둔다.
- 수집물은 내부 참고용이며 출처 추적을 위해 pin_url을 반드시 보존한다.

수집 백엔드:
- pinterest-dl 패키지(Python 3.10+)를 라이브러리로 사용한다.
  PinterestDL.with_api() 클라이언트를 기본으로 하고,
  --client chromium 옵션 시 브라우저 백엔드로 전환한다
  (pip install pinterest-dl[browser] + playwright install chromium 필요).
- 로그인 쿠키는 config/cookies.json에서 읽는다. 없으면 경고만 출력하고
  진행하되, 수집량이 목표에 못 미칠 수 있음을 로그로 알린다.
  README에 pinterest-dl login --from-browser -o config/cookies.json
  절차를 안내한다.
- pinterest-dl이 반환하는 메타데이터의 JSON 스키마는 버전에 따라
  다를 수 있으므로, 첫 응답에서 필드를 확인해 방어적으로 매핑한다.
  필수: image_url(원본 해상도 src). 있으면 저장: alt, pin_url 또는 pin_id.

입력:
config/queries.csv
  columns: axis,category_id,category_name,query,target_count
config/seed_pins.csv (선택)
  columns: axis,category_id,seed_pin_url,related_count

출력:
metadata/search_results.csv
metadata/downloads.csv
metadata/errors.csv
metadata/duplicates.csv
metadata/review_queue.csv

이미지 저장:
00_raw_downloaded/{axis}/{category_id}/{image_id}.jpg
01_deduplicated/{axis}/{category_id}/{image_id}.jpg
03_contact_sheets/{axis}_{category_id}.jpg
04_review_gallery/{axis}_{category_id}.html

image_id 규칙:
pinimg URL에서 크기 프리픽스(/236x/, /474x/, /736x/, /originals/ 등)를
제거한 경로의 sha1 해시 앞 12자리. 모든 단계에서 이 id로 조인한다.

구현 스크립트:

[scripts/01_collect.py]
- queries.csv를 읽어 각 query를 pinterest-dl search로 수집한다.
  수집 목표는 행의 target_count, --max-per-query가 있으면 override.
- seed_pins.csv가 있으면 각 시드 핀을 pinterest-dl scrape로
  related_count만큼 연관핀 수집한다 (source=related로 표기).
- 쿼리/시드 간 5~10초 랜덤 대기.
- 결과를 metadata/search_results.csv에 append 저장:
  image_id, axis, category_id, category_name, query, source(search|related),
  image_url, pin_url, alt, collected_at
- 같은 카테고리 내 동일 image_id는 저장 전에 제거한다.
- 옵션: --max-per-query, --client(api|chromium), --cookies, --only-axis,
  --only-category

[scripts/02_download_images.py]
- search_results.csv를 읽어 카테고리 내 image_id 중복을 제거한 뒤
  다운로드한다.
- 원본 URL 우선, 403/404 시 1200x → 736x 폴백 체인.
- User-Agent와 Referer(https://www.pinterest.com/)를 설정한다.
- ThreadPoolExecutor 동시성 4, 요청 간 0.3~0.8초 지터.
- PIL로 검증: 열리지 않으면 status=corrupt,
  min_width/min_height 미달은 status=too_small.
- 정상 이미지는 00_raw_downloaded/{axis}/{category_id}/{image_id}.jpg
- metadata/downloads.csv:
  image_id, axis, category_id, file_path, width, height,
  image_url, pin_url, alt, query, status
- 실패는 metadata/errors.csv에 url, 사유, 시각 기록.
- --resume: downloads.csv에 이미 있는 image_id는 건너뛴다.
- 옵션: --min-width(기본 500), --min-height(기본 500), --workers

[scripts/03_deduplicate.py]
- imagehash.phash(기본 hash_size=8, 64bit)를 사용한다.
- 같은 카테고리 내부에서 해밍 거리 threshold(기본 8) 이하를
  중복 그룹으로 묶고, 그룹에서 해상도가 가장 큰 이미지를 대표로 남긴다.
- 대표 이미지만 01_deduplicated/{axis}/{category_id}/로 복사한다.
- 제거된 이미지는 metadata/duplicates.csv에
  image_id, kept_image_id, hamming, reason=within_category 기록.
- 서로 다른 카테고리 간 pHash 일치는 삭제하지 않는다.
  duplicates.csv에 reason=multi_label로 양쪽 image_id와
  카테고리를 기록만 한다 (한 사진이 여러 축에 속하는 것은 정상).
- 옵션: --threshold

[scripts/04_rank_images.py]
- 01_deduplicated의 이미지에 대해 점수를 계산한다.
- person_score: ultralytics YOLOv8n person 클래스.
  1명 +2, 2명 이상 +1, 미검출 -3.
- clip_score: open_clip ViT-B-32 (laion2b 가중치).
  이미지 임베딩과 해당 query 텍스트 임베딩의 cosine similarity를
  카테고리 내 z-score 정규화 후 -2~+2 클리핑.
  추가로 모든 카테고리 프롬프트("a photo of {category_name} ...")와
  zero-shot 비교해 argmax가 배정 카테고리와 다르면
  category_match=False.
  디바이스는 cuda → mps → cpu 순 자동 선택.
- resolution_score: 긴 변 1200+ → +2, 800+ → +1, 500 미만 → -2
- aspect_score(w/h): 0.60~0.85 → +2, 0.85~1.10 → +1, 1.30 초과 → -1
- keyword_score: alt/URL에 query 키워드 포함 시 +1
- final_score = 2.0*clip_score + person_score + resolution_score
  + aspect_score + keyword_score
- torch/open_clip/ultralytics 미설치 시 해당 항 0으로 두고
  경고 후 나머지 점수로 진행한다 (graceful degradation).
- metadata/review_queue.csv를 생성하고 카테고리별 final_score
  내림차순으로 정렬한다.

[scripts/05_make_contact_sheets.py]
- review_queue.csv 상위 60장으로 카테고리별 contact sheet 생성.
- 한 줄 6장, 썸네일 폭 400px, 각 썸네일 아래 image_id 표기.
- 03_contact_sheets/{axis}_{category_id}.jpg (JPEG quality 85).
- 추가로 04_review_gallery/{axis}_{category_id}.html 생성:
  lazy-load 썸네일 그리드, 각 항목에 image_id / final_score /
  pin_url 링크 / approve·reject 체크박스,
  하단 버튼으로 체크 결과를 CSV로 다운로드(클라이언트 JS만 사용,
  서버 불필요). contact sheet는 회의 공유용, HTML은 검수 작업용이다.

review_queue.csv columns:
image_id, file_path, axis, category_id, category_name, query, source,
width, height, person_score, clip_score, category_match, final_score,
pin_url, review_status(pending|approved|rejected|wrong_category|
low_quality|duplicate)

requirements.txt:
pinterest-dl
pandas
requests
pillow
tqdm
imagehash
numpy

requirements-optional.txt:
torch
open_clip_torch
ultralytics
playwright

README.md:
- Python 3.10+ 요구사항과 설치 방법
- cookies.json 생성 절차 (pinterest-dl login --from-browser)
- 실행 순서와 소규모 테스트 방법
- queries.csv / seed_pins.csv 작성법
- 자주 발생하는 오류 (쿠키 만료, 403 폴백, pinterest-dl 스키마 변경)
- 수집물은 내부 참고용이며 재배포/외부 공개 금지라는 사용 범위 명시
```

## 15. 리스크와 준수 사항

솔직한 전제 위에서 진행한다.

```txt
1. ToS/robots: Pinterest robots.txt는 크롤러 접근을 전면 차단하고,
   ToS는 공식 API 외의 자동화 접근을 금지한다. 예의 있는 수집
   (지연, 회피 기능 없음)을 지키더라도 계정 제한이나 IP 차단
   가능성은 감수해야 하는 리스크다. 중요 계정으로 돌리지 않는다.

2. 저작권: 수집 이미지는 대부분 제3자 저작물이다. 내부 참고용
   열람은 위험이 낮은 편이지만, 팀 외부 공개, 발표자료 재배포,
   상업적 사용은 하지 않는다. pin_url을 메타데이터에 보존해
   출처 추적이 항상 가능하게 한다.

3. AI 학습: 내부 레퍼런스 열람과 모델 학습 투입은 다른 문제다.
   학습에 쓰려는 시점이 오면 라이선스/초상권 관점에서 별도 검토한다.
   장기적으로는 이미지 원본 대신 포즈 키포인트와 구도 메타데이터만
   추출해 보관하는 방안이 법적 노출과 저장 비용을 함께 줄인다.

4. 초상권: 식별 가능한 인물 사진이므로 위 사용 범위 제한을
   README에 명시하고 팀원에게 공유 시에도 같은 조건을 붙인다.
```

## 16. 최종 워크플로우 요약

```txt
1. queries.csv에 카테고리별 검색어 작성 (영/한/일 혼합)
2. pinterest-dl로 검색 수집, 필요 카테고리는 시드 핀 연관핀으로 보강
3. image_id(정규화 URL 해시) 기준 사전 중복 제거 후 원본 해상도 다운로드
4. pHash로 카테고리 내 중복 제거, 교차 축은 멀티라벨로 기록
5. person detector + CLIP + 해상도/비율 점수로 정렬,
   wrong_category 자동 플래그
6. contact sheet(회의용) + HTML 갤러리(검수용) 생성
7. 사람이 최종 검수
```

가장 중요한 판단은 v1과 같다: 처음부터 완벽한 데이터셋을 만들려 하지 않고, 자동으로 많이 모은 뒤 자동 필터로 줄이고, 최종 판단은 사람이 한다. v2에서 달라진 것은 그 자동 수집이 실제로 돌아가게 만드는 전제 조건(로그인 세션, API 기반 원본 URL, 검증된 도구)을 명시한 것이다.

## 부록: Playwright 직접 구현 스펙 (최후 폴백)

pinterest-dl이 API 변경으로 깨졌고 브라우저 백엔드도 실패할 때만 사용한다. v1 스펙에 두 가지 수정을 반영한다.

```txt
- Chromium persistent context 사용. 프로필 디렉토리에 최초 1회
  수동 로그인 필수 (headless=False로 열어 로그인 후 종료).
  비로그인 상태에서는 로그인 벽으로 인해 수십 장 이상 수집 불가.
```
